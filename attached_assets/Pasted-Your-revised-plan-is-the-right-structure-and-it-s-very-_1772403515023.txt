Your revised plan is the right structure and it’s very close to implementation-ready. Before you lock it in, there are a few concrete fixes/holes (based on what’s actually in the repo right now) that you should bake into the tasks so you don’t hit “why is this page still broken?” whack-a-mole.

The plan is solid — here are the must-fix corrections
1) T001 schema changes: expand + correct a few fields to match the UI

People table: your list is good, but the UI also writes these right now:

star_pattern (TEXT)

star_intensity (INTEGER)

star_flare_count (INTEGER)

Family settings table: Settings UI expects these columns (currently not in schema.sql):

tagline (TEXT)

admin_emails (TEXT[])

planner_emails (TEXT[])

Join requests table: Settings UI writes this on approval:

reviewed_by_email (TEXT) (optional but recommended: reviewed_at TIMESTAMPTZ)

Users table: you have two choices—pick one and align code to it:

Option A (recommended): add role TEXT DEFAULT 'user' so user.role === 'admin' works (Settings checks this today).

Option B: don’t add role; make “admin” purely driven by family_settings.admin_emails, and remove the user.role check from Settings.

Calendar events table: your proposed columns don’t match the Calendar page.
Calendar expects:

title, description

date (the code uses event.date)

event_type

person_ids UUID[]

is_recurring BOOLEAN

color TEXT

created_at
So adjust that table definition accordingly.

Trip participants constraint change: “fix status CHECK to include invited” is correct, but note:

You can’t “ALTER CHECK” inline. You’ll need to DROP the existing constraint and recreate it.

2) T003 “functions route” path needs to match your base44 client

Your base44Client.js calls:
POST /api/functions/:functionName
So implement that route shape (not /api/functions/invoke).

Also: Home does const { data } = await base44.functions.invoke(...), so your function endpoints should return:

{ "data": ... }

(or update the frontend to not destructure { data }).

3) Phase 2 security: add one more hard requirement

Your T004 whitelist work is essential, but it’s not complete unless you also:

whitelist filter columns (because GET /filter currently interpolates column names too)

whitelist create/update columns (because POST and PATCH accept arbitrary keys today)

If you don’t whitelist those, you’ll still have injection and “write-any-column” problems even after entity whitelisting.

4) Dangerous admin functions need environment gating

cleanupTestData is a foot-gun even with a confirm dialog.

Add a hard rule:

only runs when NODE_ENV !== 'production' (or ALLOW_DESTRUCTIVE_ADMIN=true)

and only for admins

That prevents a “one bad click nuked prod” incident.
